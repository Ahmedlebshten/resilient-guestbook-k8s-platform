## Critical Infrastructure Policies

### Compute
- Use a **single AWS EC2 instance** (`t3.micro`) as the unified K3s Control + Worker plane.
- The instance runs **Ubuntu 24.04 LTS** (Noble Numbat).

### Networking & DNS
- Use **DuckDNS** for free Dynamic DNS to handle Public IP changes after EC2 stop/start cycles.
- Terraform auto-generates `ansible/inventory.ini` via a `local_file` resource to ensure zero manual IP updates.

### Container Images
- All Docker images are built with `--no-cache --pull` to prevent stale layer caching.
- Images are pushed to **Docker Hub** under the `ahmedlebshten/guestbook` repository.
- The Kubernetes Deployment uses `imagePullPolicy: Always`.

### Database High Availability
- Deploy the **CloudNativePG (CNPG)** Operator.
- Provision a **3-node PostgreSQL 15** cluster with automatic self-healing and failover.
- Database credentials are managed via Kubernetes Secrets (auto-generated by the CNPG Operator).

### Security
- All application pods should use `securityContext` where applicable.
- Container image vulnerabilities are scanned by **Trivy** in the CI pipeline.
- SSH private keys and Terraform state files are excluded from Git via `.gitignore`.

### GitOps
- **ArgoCD** watches the `k8s-manifests/` directory for changes.
- The CI pipeline updates the image tag in `app-deployment.yml` via `sed`, commits, and pushes.
- ArgoCD automatically syncs the cluster to match the Git state.

## Key Commands Reference

### Terraform
```bash
cd terraform/
terraform init          # Initialize providers
terraform plan          # Preview changes
terraform apply         # Provision infrastructure
terraform destroy       # Tear down infrastructure
```

### K3s Setup (via SSH)
```bash
scp -i terraform/k3s-key ansible/setup_k3s.sh ubuntu@<EC2_IP>:~/
ssh -i terraform/k3s-key ubuntu@<EC2_IP> "chmod +x setup_k3s.sh && ./setup_k3s.sh"
```

### Kubernetes Operations (on EC2)
```bash
kubectl apply -f k8s-manifests/             # Deploy all manifests
kubectl get pods                            # Check pod status
kubectl get svc                             # Check services
kubectl logs -f deployment/guestbook-app    # Stream app logs
kubectl rollout restart deployment/guestbook-app  # Force pod restart
```

### Database Backup (on EC2)
```bash
chmod +x scripts/db-backup.sh
./scripts/db-backup.sh
# Output: /home/ubuntu/db_backups/full_backup_<timestamp>.sql
```

### Manual Docker Build & Push
```bash
export DOCKER_USERNAME="your_dockerhub_username"
chmod +x scripts/build_and_push.sh
./scripts/build_and_push.sh
```

### DuckDNS IP Update
```bash
curl "https://www.duckdns.org/update?domains=ahmed-guestbook&token=YOUR_TOKEN&ip=<NEW_IP>"
```

### Monitoring Setup (on EC2)
```bash
chmod +x monitoring/setup_monitoring.sh
./monitoring/setup_monitoring.sh
# Grafana: http://<EC2_IP>:30002 (admin / auto-generated password)
```

## Application Logic

The **Guestbook** application is a server-side rendered Express.js web app:

1. On startup (`app.listen`), the app calls `initDB()` which creates the `messages` table if it doesn't exist.
2. `GET /` queries all messages from PostgreSQL, renders them as HTML using TailwindCSS (via CDN), and returns the page.
3. `POST /send` inserts a new message (name + message) into the database and redirects back to `/`.
4. Database connection parameters (`DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_NAME`) are injected via Kubernetes environment variables and Secrets.
